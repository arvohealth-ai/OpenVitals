# OpenVitals / OpenClaw Credentials Checklist

> Languages: [English](./credentials-setup.md) | [简体中文](./credentials-setup-zh.md)

Last reviewed: 2026-04-26.

This guide lists the credentials, permissions, and local secrets required to
run real OpenClaw + OpenVitals E2E, Oura, WHOOP, and iPhone + Apple Watch
hardware tests in this repository. Never commit real secrets to Git. The
repository ignores `.env` and `.env.*`, but that does not make accidental
leakage safe.

For the end-to-end field workflow covering setup, OMX multi-agent execution,
OpenRouter, Oura / WHOOP connectivity, OpenClaw E2E, Xcode / iOS / watchOS
device debugging, and evidence capture, see the
[OpenVitals Field Runbook](./openvitals-field-runbook.md).

## 0. Overview

Collect these first:

| Category | Required items | Used for |
| --- | --- | --- |
| OpenVitals local runtime | `OPENVITALS_ADMIN_TOKEN`, `OPENVITALS_SECRETS_KEY` | live bootstrap, encrypted provider credentials |
| OpenVitals agent token | `OPENVITALS_AGENT_TOKEN` | calling API / MCP; generated by bootstrap, not requested from a third party |
| OpenClaw gateway | `OPENCLAW_GATEWAY_TOKEN` | shared-secret auth if you run OpenClaw gateway / control UI |
| Model provider | at least one of `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GEMINI_API_KEY` / `GOOGLE_API_KEY`, `OPENROUTER_API_KEY` | model access for OpenClaw agent loop / workers |
| Oura | `OPENVITALS_OURA_CLIENT_ID`, `OPENVITALS_OURA_CLIENT_SECRET`, `OPENVITALS_OURA_REDIRECT_URI` | Oura OAuth + API |
| WHOOP | `OPENVITALS_WHOOP_CLIENT_ID`, `OPENVITALS_WHOOP_CLIENT_SECRET`, `OPENVITALS_WHOOP_REDIRECT_URI`, `OPENVITALS_WHOOP_WEBHOOK_SECRET` | WHOOP OAuth + webhook |
| Apple Health / Apple Watch | Apple Developer Team, bundle IDs, HealthKit capability, device permissions | device-side iPhone / Watch ingest; no cloud API key exists |
| Android Health Connect | Android app package / signing + Health Connect permissions | Android smoke path; no cloud API key exists |

## 1. Store secrets locally

Create a local-only env file in the repo root:

```bash
cd <path-to-openvitals>
touch .env.live.local
chmod 600 .env.live.local
```

If the file contains strict shell `export KEY=value` lines, you can load it
like this:

```bash
set -a
source .env.live.local
set +a
```

Do not blindly `source .env`. Dotenv syntax and shell syntax are not identical;
values with spaces, quotes, comments, or non-shell formatting may break.
Prefer the project's dotenv loader in scripts, and export only the variables you
need in ad hoc terminal sessions.

Generate a strong local secret:

```bash
openssl rand -hex 32
```

`OPENVITALS_SECRETS_KEY` encrypts stored provider credentials. Once you start
using real Oura / WHOOP credentials, keep it stable. Rotating it casually will
make old credentials undecryptable and may force users through OAuth again.

## 2. Core OpenVitals variables

```bash
export OPENVITALS_MODE=live
export OPENVITALS_DB_PATH=.openvitals/openvitals.sqlite
export OPENVITALS_ADMIN_TOKEN="<openssl-rand-hex-32>"
export OPENVITALS_SECRETS_KEY="<openssl-rand-hex-32>"
```

You do not request `OPENVITALS_AGENT_TOKEN` ahead of time. Start the API and
generate it with live bootstrap:

```bash
OPENVITALS_MODE=live pnpm --filter @openvitals/api demo

curl -X POST "http://127.0.0.1:3000/v1/live/bootstrap" \
  -H "x-openvitals-admin: $OPENVITALS_ADMIN_TOKEN" \
  -H "content-type: application/json" \
  -d '{"userId":"user_live","name":"Live User","timezone":"Asia/Shanghai","providers":["apple-health","oura","whoop"],"createTokens":true}'
```

Save the derived token from the response:

```bash
export OPENVITALS_AGENT_TOKEN="<derived-token-from-bootstrap>"
```

## 3. OpenClaw gateway and model providers

### 3.1 OpenClaw gateway token

If you only run `pnpm openclaw:e2e`, you do not need a gateway token. If you
run OpenClaw gateway / control UI, generate a long random token:

```bash
export OPENCLAW_GATEWAY_TOKEN="$(openssl rand -hex 32)"
```

OpenClaw documentation describes gateway shared-secret auth as
`gateway.auth.token` / `gateway.auth.password`; this repository also accepts
`OPENCLAW_GATEWAY_TOKEN` / `OPENCLAW_GATEWAY_PASSWORD`.

### 3.2 Model keys: at least one

The OpenClaw long-running agent loop needs a model provider. One key is enough
to get started. `OPENAI_API_KEY` or `ANTHROPIC_API_KEY` is the most pragmatic
default.

#### OpenAI

References:

- API keys: <https://platform.openai.com/api-keys>
- Quickstart: <https://platform.openai.com/docs/quickstart>

Steps:

1. Sign in to OpenAI Platform.
2. Select the correct organization / project.
3. Open **API Keys** and create a new secret key.
4. A name like `openvitals-openclaw-local` is fine.
5. Save the key while it is visible.
6. Export it:

```bash
export OPENAI_API_KEY="<openai-secret-key>"
```

The official docs treat the key as a secret. Keep it in environment variables or
server-side secret storage. Never expose it in browser code or mobile clients.

#### Anthropic Claude

References:

- Console: <https://console.anthropic.com/>
- API overview: <https://platform.claude.com/docs/en/api/overview>

Steps:

1. Sign in to Anthropic Console.
2. Configure billing / workspace.
3. Create a key under account settings or API keys.
4. Save it.
5. Export it:

```bash
export ANTHROPIC_API_KEY="<anthropic-api-key>"
```

Anthropic API requests use `x-api-key`; OpenClaw and the SDK typically read
`ANTHROPIC_API_KEY`.

#### Google Gemini

References:

- API key management: <https://aistudio.google.com/app/apikey>
- Gemini API docs: <https://ai.google.dev/api>

Steps:

1. Sign in to Google AI Studio.
2. Create or select a Google Cloud project.
3. Create an API key.
4. Restrict the key and add budget alerts in Google Cloud where possible.
5. Export it:

```bash
export GEMINI_API_KEY="<gemini-api-key>"
# Some tools only read GOOGLE_API_KEY, so set both when needed:
export GOOGLE_API_KEY="$GEMINI_API_KEY"
```

Gemini REST requests use the `x-goog-api-key` header.

#### OpenRouter

References:

- API keys: <https://openrouter.ai/settings/keys>
- Authentication: <https://openrouter.ai/docs/api-reference/authentication>
- Chat completions: <https://openrouter.ai/docs/api/api-reference/chat/send-chat-completion-request>
- Models endpoint: <https://openrouter.ai/docs/api/api-reference/models/get-models>

Steps:

1. Sign in to OpenRouter.
2. Create an API key.
3. Set a credit limit so long-running agents cannot run away silently.
4. Export it:

```bash
export OPENROUTER_API_KEY="<openrouter-api-key>"
export OPENVITALS_LLM_PROVIDER=openrouter
export OPENVITALS_OPENROUTER_API_URL="https://openrouter.ai/api/v1"

# Optional: when empty, `pnpm llm:openrouter:smoke` selects the cheapest priced
# text chat model from /models.
export OPENVITALS_OPENROUTER_MODEL=""
export OPENVITALS_OPENROUTER_MAX_TOKENS=8
export OPENVITALS_OPENROUTER_MAX_ATTEMPTS=8
export OPENVITALS_OPENROUTER_ALLOW_FREE=true
```

The standard variable is `OPENROUTER_API_KEY`; OpenVitals also accepts the local
compatibility alias `OPEN_ROUTER_API_KEY`.

OpenRouter uses Bearer tokens. The smoke command calls the OpenAI-compatible
`POST /chat/completions` endpoint. After env setup:

```bash
pnpm llm:openrouter:smoke
```

The command loads `.env`, `.env.local`, and `.env.live.local`, confirms the key
exists without printing it, ranks candidate text chat models by price, retries
the cheapest available candidates, and prints only the chosen model, a short
reply preview, and token usage. To skip free endpoints:

```bash
OPENVITALS_OPENROUTER_ALLOW_FREE=false pnpm llm:openrouter:smoke
```

## 4. Oura Ring OAuth / API

References:

- Oura API docs: <https://cloud.ouraring.com/docs/>
- Oura OAuth docs: <https://cloud.ouraring.com/docs/authentication>
- Oura applications: <https://cloud.ouraring.com/oauth/applications>

Required env:

```bash
export OPENVITALS_OURA_CLIENT_ID="<oura-client-id>"
export OPENVITALS_OURA_CLIENT_SECRET="<oura-client-secret>"
export OPENVITALS_OURA_REDIRECT_URI="https://<your-domain>/v1/connect/callback/oura"
export OPENVITALS_OURA_API_URL="https://api.ouraring.com"
export OPENVITALS_OURA_SCOPE="personal daily heartrate workout spo2"
```

For local testing without public HTTPS, expose the API with ngrok or Cloudflare
Tunnel and use the tunnel URL in the app's redirect URI:

```bash
export OPENVITALS_OURA_REDIRECT_URI="https://<tunnel-host>/v1/connect/callback/oura"
```

Steps:

1. Sign in to `cloud.ouraring.com` with the Oura account.
2. Open **My Applications**.
3. Create an OAuth application.
4. Set the redirect URI exactly equal to `OPENVITALS_OURA_REDIRECT_URI`.
5. Copy the client ID and client secret.
6. Configure scopes.

Relevant official scopes:

- `email`: optional, usually unnecessary here
- `personal`: age / sex / height / weight
- `daily`: daily sleep / activity / readiness
- `heartrate`: Gen 3 heart-rate time series
- `workout`: auto-detected and user-entered workouts
- `spo2`: nightly SpO2 summaries

Recommended scope set:

```bash
personal daily heartrate workout spo2
```

Notes:

- Authorization URL: `https://cloud.ouraring.com/oauth/authorize`
- Token URL: `https://api.ouraring.com/oauth/token`
- Oura uses exact redirect URI matching.
- Oura cloud data is delayed, provider-mediated data, not continuous raw sensor
  streaming.
- Oura heart-rate rows should become `sample`, not `live_signal`, in
  OpenVitals.

## 5. WHOOP OAuth / API

References:

- WHOOP Developer Dashboard: <https://developer.whoop.com/>
- Getting started: <https://developer.whoop.com/docs/developing/getting-started/>
- OAuth docs: <https://developer.whoop.com/docs/developing/oauth/>
- API reference: <https://developer.whoop.com/api/>

Required env:

```bash
export OPENVITALS_WHOOP_CLIENT_ID="<whoop-client-id>"
export OPENVITALS_WHOOP_CLIENT_SECRET="<whoop-client-secret>"
export OPENVITALS_WHOOP_REDIRECT_URI="https://<your-domain>/v1/connect/callback/whoop"
export OPENVITALS_WHOOP_WEBHOOK_SECRET="<openssl-rand-hex-32>"
export OPENVITALS_WHOOP_SCOPE="offline read:sleep read:recovery read:workout read:cycles read:profile read:body_measurement"
```

For local testing, use an HTTPS tunnel if the dashboard refuses
`http://127.0.0.1`:

```bash
export OPENVITALS_WHOOP_REDIRECT_URI="https://<tunnel-host>/v1/connect/callback/whoop"
```

Steps:

1. Sign in to the WHOOP Developer Dashboard.
2. Create a team.
3. Create an app.
4. Configure a redirect URI that exactly matches the OAuth request.
5. Configure scopes. Cover sleep / recovery / workout at a minimum; adding
   cycles / profile / body measurement makes richer explanations possible.
6. Save the client ID and client secret.
7. If you configure webhooks, generate a local
   `OPENVITALS_WHOOP_WEBHOOK_SECRET`. In this repository it is currently used as
   a dev / local shared-secret check, not as a claim of official production
   signature verification.

Official endpoints:

- Authorization URL: `https://api.prod.whoop.com/oauth/oauth2/auth`
- Token URL: `https://api.prod.whoop.com/oauth/oauth2/token`
- API base: `https://api.prod.whoop.com/developer/v2`

Relevant official scopes:

- `read:recovery`
- `read:cycles`
- `read:workout`
- `read:sleep`
- `read:profile`
- `read:body_measurement`
- `offline` for refresh tokens

Notes:

- WHOOP cloud API is provider-mediated recovery / sleep / workout / cycle
  summary data, not continuous raw HR streaming.
- For the smallest viable path, `offline read:sleep read:recovery read:workout`
  is enough.
- For richer daily brief / explanation outputs, request the broader recommended
  scope set.

Development fallback:

```bash
export OPENVITALS_WHOOP_ACCESS_TOKEN="<temporary-dev-access-token>"
export OPENVITALS_WHOOP_BRIDGE_URL="<optional-bridge-url>"
```

Use these only for debugging or migration. The real path should still be
per-user OAuth.

## 6. Apple Health / Apple Watch

HealthKit does not have a server-side cloud API key. The OpenVitals Apple path
is device-side ingest: an iPhone / Watch app reads HealthKit and uploads to the
local API.

References:

- HealthKit overview: <https://developer.apple.com/documentation/healthkit>
- Setup guide: <https://developer.apple.com/documentation/healthkit/setting-up-healthkit>
- `requestAuthorization`: <https://developer.apple.com/documentation/healthkit/hkhealthstore/requestauthorization(toshare:read:)>
- `HKWorkoutSession`: <https://developer.apple.com/documentation/healthkit/hkworkoutsession>
- `HKLiveWorkoutBuilder`: <https://developer.apple.com/documentation/healthkit/hkliveworkoutbuilder>

You need:

| Item | How to get it |
| --- | --- |
| Apple Developer Team ID | Apple Developer account / Xcode account |
| iOS bundle ID | Apple Developer portal or Xcode automatic signing |
| watchOS bundle ID | only required for the optional Apple Watch live-workout path |
| HealthKit capability | Xcode target -> Signing & Capabilities -> `+ Capability` -> HealthKit |
| Info.plist usage strings | `NSHealthShareUsageDescription`, `NSHealthUpdateUsageDescription` |
| Physical device | iPhone; paired Apple Watch for live workout HR |

Requested HealthKit read permissions should include:

- Heart rate
- HRV SDNN
- Resting heart rate
- Step count
- Sleep analysis
- Workouts

Important Apple platform rules:

- Enable the HealthKit capability before using HealthKit.
- Call `HKHealthStore.requestAuthorization(toShare:read:)` for the data types.
- Configure HealthKit usage descriptions or the app will crash on permission
  request.
- Apple Watch live HR requires the active workout path:
  `HKWorkoutSession` + `HKLiveWorkoutBuilder`.

OpenVitals does not need a shared secret here. It needs provenance in the upload
payload:

- `HKSourceRevision.bundleIdentifier`
- device info
- timezone
- unit
- HealthKit sample UUID / source record hash
- anchor state
- `captureMode="mirrored"` plus source metadata for Oura / WHOOP samples
  mirrored into Apple Health

## 7. Android Health Connect

In v0.6, Android Health Connect is a follow-on smoke path after the iOS path is
green. There is no cloud API key. You need app permissions and user consent:

- Android app package name
- debug / release signing key
- Health Connect permissions for the smoke-test metrics
- Android device or emulator with Health Connect available

Preserve source / provenance / anchor-like sync metadata here as well. Do not
describe Health Connect samples as a cloud live stream.

## 9. Recommended verification order

1. Start with one model key and run:

```bash
pnpm openclaw:e2e
```

2. Set the OpenVitals core env, start the live API, run bootstrap, and save
   `OPENVITALS_AGENT_TOKEN`.
3. Configure the Oura app and env, then run Oura connect/start + callback +
   sync.
4. Configure the WHOOP app and env, then run WHOOP connect/start + callback +
   sync.
5. Run Apple Health historical ingest through the iPhone collector.
6. Run Apple Watch live HR through the workout collector.
7. Enable Oura / WHOOP writes into Apple Health, upload mirrored samples, and
   verify dedupe.

Do not describe Oura or WHOOP cloud data as continuous raw sensor streams. Only
the Apple Watch active workout collector path qualifies as a live signal.
